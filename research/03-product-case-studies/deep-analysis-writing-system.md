# 深度游戏拆解写作系统

本文档用于把“同类产品分析”从简短观察升级为可复用的深度拆解写作流程。它以 `assets/标准化游戏拆解案框架.md` 的八大模块为主结构，并参考 `assets/` 中的拆解示例形成仓库内的标准写法。

本系统的目标不是写游戏测评，而是逆向还原一款游戏的设计逻辑、系统联动、商业闭环、玩家体验和可迁移假设，最终服务本项目的核心玩法判断。

## 适用场景

当用户提到某个产品时，先判断进入哪种深度：

| 深度 | 适用情况 | 输出位置 |
| --- | --- | --- |
| 市场参照 | 只是顺手提到一个相似产品或品类方向 | `game-design-workflow/market-reference.md` |
| 轻量案例 | 需要快速记录核心循环、机制和启发 | `research/03-product-case-studies/`，使用 `research/templates/product-case-template.md` |
| 专项拆解 | 只深挖某个系统，如基建、战斗、养成、经济 | `research/03-product-case-studies/`，使用深度模板并标注专项范围 |
| 完整深度拆解 | 需要形成可给立项或设计讨论使用的长文 | `research/03-product-case-studies/`，使用 `research/templates/deep-product-analysis-template.md` |
| 横向比较 | 同时比较多个产品的结构差异 | `research/04-cross-game-comparisons/` |

默认策略：信息不足时先写轻量案例；用户明确要求“深度拆解”“完整分析”“作为标准案例”时，使用深度拆解。

## 写作总流程

1. 确定拆解问题
   - 这次拆解要回答什么设计问题？
   - 它与本项目的唯一核心卡、组合、费用取舍、动态平衡或回合制战斗有什么关系？

2. 选择拆解深度
   - 完整产品拆解：覆盖八大模块。
   - 专项系统拆解：保留八大模块顺序，但只对相关模块展开，其他模块写“与本次问题关系较弱”。

3. 收集资料
   - 官方资料：官网、公告、开发者访谈、规则说明、商店页。
   - 玩家资料：社区反馈、攻略、评价、长线玩家讨论。
   - 自体验资料：试玩记录、流程截图、系统观察。
   - 数据资料：上线时间、平台、评分、留存/流水/销量等公开数据。

4. 写八大模块
   - 先写结论，再写证据。
   - 每个模块都要回答“它如何支撑玩家循环或商业目标”。
   - 核心循环和系统架构必须使用 Mermaid 图或文本流程图。

5. 提炼可迁移结论
   - 不写“这个游戏很好”。
   - 必须写清本项目可借鉴的结构、本项目应避免的雷同和可验证设计假设。

6. 流转到后续文档
   - 成熟观察进入 `research/05-design-hypotheses/`。
   - 可执行玩法方向进入 `game-design-workflow/idea-proposals/`。
   - 仍不确定的疑问写回 `research/00-index-and-roadmap/current-questions.md`。

## 八大模块的仓库化改造

原框架的八大模块保持顺序不变，但每个模块在本仓库里增加一个“项目转化”要求：

| 模块 | 写作重点 | 必须转化的问题 |
| --- | --- | --- |
| 1. 核心定位、基础信息、商业大盘 | 产品是什么、服务谁、如何赚钱 | 它是否与本项目争夺同类玩家心智？ |
| 2. 全局玩家体验与设计目标 | 玩家从进入到留存的真实体感 | 它主要满足哪种情绪价值？本项目是否需要同类价值？ |
| 3. 核心玩法循环 | 登录、行为、产出、消耗、成长、回流 | 哪个循环结构可迁移到唯一核心卡对战？ |
| 4. 全链路架构 | 战斗、成长、资源、探索、叙事、社交如何联动 | 哪些系统联动会增强或稀释核心战斗？ |
| 5. 内容与关卡体系 | 内容如何填充时间、制造阶段目标 | 第一阶段原型需要借鉴哪种内容节奏？ |
| 6. 数值与经济闭环 | 资源产出、消耗、回收、付费卡点 | 费用、组合和核心卡价值是否能形成健康闭环？ |
| 7. 叙事与视听包装 | 世界观、角色、视觉、音频如何留人 | 核心卡身份是否需要叙事包装支撑？ |
| 8. 优劣复盘与优化方案 | 可复用亮点、真实短板、落地改法 | 哪些观察应变成假设、提案或暂缓问题？ |

## 文件命名

建议使用英文小写、数字和连字符：

```text
research/03-product-case-studies/<product-name>-deep-analysis.md
research/03-product-case-studies/<product-name>-<system-name>-deep-analysis.md
```

示例：

```text
research/03-product-case-studies/mi-chang-sheng-deep-analysis.md
research/03-product-case-studies/arknights-endfield-base-system-deep-analysis.md
```

如果需要按日期区分版本，可在文件名前加 `YYYY-MM-DD-`。

## 资料可信度规则

- 最新上线时间、版本内容、商业数据、评分和公开市场表现必须联网核实并标注来源。
- 无法核实的数据要写成“公开资料未确认”或“社区估计”，不要当作事实。
- 玩家评价要区分样本来源，如 Steam、TapTap、Bilibili、贴吧、Reddit、Discord。
- 用户给出的本地素材可以作为写作参考，但不能替代可核验来源。

## 图示规范

深度拆解至少包含两张图：

1. 核心循环图：放在模块 3。
2. 系统关系图：放在模块 4。

优先使用 Mermaid，便于 Markdown 维护。例如：

```mermaid
flowchart LR
    Login[登录] --> Action[核心行为]
    Action --> Output[产出]
    Output --> Spend[消耗]
    Spend --> Growth[成长/能力提升]
    Growth --> Unlock[解锁新内容]
    Unlock --> Return[次日回流]
```

## 质量检查

交付前检查：

- 是否明确写出本次拆解目的。
- 是否区分事实、推断和个人判断。
- 是否保留八大模块顺序。
- 是否包含核心循环图和系统关系图。
- 是否每个展开模块都有“对本项目的启示”。
- 是否输出 3-5 条可迁移亮点、3-5 条应避免问题。
- 是否至少形成一个可验证设计假设，或解释为什么暂不形成假设。
- 是否没有把竞品系统直接复制成本项目设定。

## 输出状态

深度拆解完成后，状态仍然是 Research，不是 Accepted。

只有当拆解结论被转化为设计假设、提案、评估和拟修改，并经用户确认后，才可能进入 `game-design-workflow/core-concept.md`。
