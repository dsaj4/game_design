# 深度拆解迭代协议

本文档定义深度拆解报告的迭代方式，避免评审只停留在“好/不好”的主观判断。

## 迭代触发条件

满足任一条件时必须迭代：

- 总分低于 50 / 60。
- 任一维度低于 3 分。
- “核心循环”“系统联动”“本项目转化”任一项低于 4 分。
- 评审指出事实错误、来源不明或关键推断缺证据。
- 报告没有产出可验证设计假设，也没有解释暂不产出假设的理由。

## 迭代优先级

1. 事实错误和来源问题。
2. 核心循环不闭环。
3. 系统联动只罗列不解释。
4. 本项目转化不可验证。
5. 语言、排版和表达密度。

## 迭代动作类型

| 类型 | 说明 | 输出 |
| --- | --- | --- |
| Report Fix | 修改具体报告内容 | 更新 `research/03-product-case-studies/` 中的报告 |
| Template Fix | 模板缺少必要字段 | 更新 `research/templates/deep-product-analysis-template.md` |
| System Fix | 流程规则不清晰 | 更新 `research/03-product-case-studies/deep-analysis-writing-system.md` |
| Rubric Fix | 评分标准无法指导判断 | 更新本目录评审文档 |
| Follow-up Hypothesis | 观察已足够转化 | 新建或更新 `research/05-design-hypotheses/` |

## 迭代记录格式

每轮迭代在 `iteration-logs/` 中创建文件：

```text
I-YYYY-MM-DD-<report-name>.md
```

必须记录：

- 评审对象。
- 初始总分与关键低分项。
- 本轮修改了哪些文件。
- 每个修改对应哪个低分项。
- 迭代后仍然保留的问题。
- 下一轮建议。

## 完成判定

一份报告可以标记为“样例可用”时，需要满足：

- 总分不低于 50 / 60。
- 没有任何维度低于 4 分。
- 关键三项不低于 4 分：核心循环、系统联动、本项目转化。
- 已完成一次评审记录和一次迭代记录。
- 报告中的事实来源清楚，未确认信息被单独列出。
