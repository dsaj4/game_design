# 深度拆解报告评审标准

本标准用于判断 Markdown 深度拆解报告是否接近 `assets/` 中高质量 docx 示例的完成度。满分 60 分，共 12 个维度，每项 1-5 分。

## 评分维度

| 维度 | 1 分 | 3 分 | 5 分 |
| --- | --- | --- | --- |
| 1. 拆解对象与范围定位 | 只是在介绍游戏，未说明拆解重点 | 说明了游戏类型、目标玩家和主要系统，但范围仍偏散 | 明确锁定拆解对象、版本、系统边界、参考价值和本报告要回答的问题 |
| 2. 基础事实与证据质量 | 依赖主观印象，无数据、版本或来源 | 有部分事实或社区反馈，但引用不稳定 | 关键信息有来源或可验证依据，能区分事实、推断和个人判断 |
| 3. 核心体验与设计目标还原 | 只写“好玩/不好玩” | 能描述玩家体验和设计目的，但缺少因果关系 | 能还原“设计目标 -> 玩家行为 -> 情绪反馈 -> 留存/付费/内容消耗”的链路 |
| 4. 核心循环拆解 | 只列玩法内容，没有循环 | 有循环图或步骤，但每步目的不清 | 清晰拆出输入、行为、消耗、产出、反馈、解锁与回流，并解释循环能否长期成立 |
| 5. 系统架构与联动关系 | 系统孤立罗列 | 能拆主要系统，但联动关系弱 | 每个系统都说明核心作用、规则、产出/消耗、与其他系统的联动和设计亮点 |
| 6. 数值、资源与经济闭环 | 只列货币和材料名称 | 能说明主要产出和消耗 | 能分析资源分层、产出-消耗-回收、成长曲线、风险点和商业化影响 |
| 7. 内容、关卡与节奏结构 | 只写有哪些内容 | 能按阶段整理内容 | 能说明内容如何承接核心循环、如何分层、何处产生疲劳、断层或目标缺失 |
| 8. 叙事、美术与题材服务玩法 | 只评价包装风格 | 能描述世界观、角色或视觉风格 | 能说明题材、叙事、角色、美术如何支撑玩法理解、目标感和情绪价值 |
| 9. 优缺点诊断质量 | 优缺点空泛，如“内容丰富”“数值不好” | 能指出问题，但缺少原因或影响 | 每个优缺点都能对应具体系统、玩家行为、影响指标和形成原因 |
| 10. 改进方案可落地性 | 只有方向，如“优化体验” | 有改法，但成本、边界不清 | 每条改进都说明改哪里、怎么改、为什么有效、可能代价和验证信号 |
| 11. 对本项目的转化价值 | 与本仓库项目无关，只是泛泛借鉴 | 能提出可参考点，但未转成设计假设 | 能转化为本项目的设计假设、提案方向、风险提醒或原型验证问题 |
| 12. Markdown 交付质量 | 层级混乱、长段堆叠、难以检索 | 结构基本清晰，但表格、图示或结论前置不足 | 标题层级稳定，结论前置，表格、流程图、系统图使用得当，可直接归档和复盘 |

## 总分建议

| 总分 | 判断 |
| --- | --- |
| 50-60 | 高质量深度拆解。可作为研究案例沉淀，并能直接转化为设计假设或提案。 |
| 40-49 | 合格拆解。结构完整，但仍需补强证据、系统联动或项目转化。 |
| 30-39 | 初级拆解。信息量足够，但更像资料整理，尚未形成设计判断。 |
| 30 以下 | 不建议归档为正式拆解。应退回补范围、证据、核心循环和结论。 |

## 样例可用门槛

一份报告可以作为仓库样例时，需要满足：

- 总分不低于 50 / 60。
- 没有任何维度低于 4 分。
- 核心循环拆解、系统架构与联动关系、对本项目的转化价值三项都不低于 4 分。
- 至少产出 1 条可验证设计假设，或明确说明暂不形成假设的理由。
- 已完成一次评审记录和一次迭代记录。

## 常见质量缺口

1. 把“系统清单”当成“系统拆解”：只列功能名称，没有说明该系统在循环中承担什么输入、消耗、产出和反馈。
2. 只写体验感受，不还原设计因果：例如“沉浸感强”“玩法丰富”，但没有解释是哪条规则、哪种资源压力、哪个反馈节点造成的。
3. 优点和缺点没有落到玩家行为：高质量缺点应能说清玩家在哪一步卡住、为什么卡住、会导致流失、疲劳、付费抵触还是目标断层。
4. 改进方案过大或过虚：“增加内容”“优化数值”“加强引导”不够，必须写到具体系统位置、改动方式、预期收益和验证信号。
5. 对本项目转化不足：报告最后如果没有回答“这会如何改变唯一核心卡 2D 回合制卡牌对战的设计判断”，就只能算产品读书笔记。

## 本项目转化专项评分

本仓库当前项目是“唯一核心卡驱动的 2D 回合制卡牌对战”。任何深度拆解报告的转化部分，应单独按以下 5 项评分，每项 1-5 分。

| 转化维度 | 1 分 | 3 分 | 5 分 |
| --- | --- | --- | --- |
| 核心相关性 | 只说“可以借鉴” | 能对应到卡牌、回合制、构筑或资源管理 | 明确对应唯一核心卡、辅助卡、费用、组合、动态平衡中的至少一项 |
| 结构转译能力 | 直接照搬原游戏系统 | 能抽象出机制结构，但还偏原产品语境 | 能转译为本项目可用的规则、问题或假设，而不是复制题材和包装 |
| 第一阶段适配性 | 提出超出当前范围的大系统 | 基本适合，但仍夹带交易、AI、地图等暂缓内容 | 严格服务第一阶段战斗原型，不扩展到暂缓系统 |
| 验证可操作性 | 没有验证方式 | 有验证方向，但不够小 | 能提出最小纸面/原型验证：玩家做什么、观察什么、成功/失败信号是什么 |
| 风险与边界意识 | 只写收益 | 能提到风险 | 能说明该借鉴可能破坏核心卡稳定性、组合层平衡、回合节奏或玩家理解成本的风险 |

转化部分建议结论格式：

```md
## 对本项目的转化

### 可借鉴结构

- ...

### 不应照搬的部分

- ...

### 可转化设计假设

- H：如果将……用于唯一核心卡/辅助卡/费用/组合规则，那么玩家会……，因为……

### 最小验证方式

- 原型范围：
- 玩家行为：
- 成功信号：
- 失败信号：

### 当前状态

Research / Design Hypothesis / Proposal Candidate / Parked
```
